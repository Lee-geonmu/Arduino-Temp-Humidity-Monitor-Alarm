# 온습도 모니터링 및 경보 시스템

DHT11 온습도 센서 기반 실시간 모니터링 + 임계값 경보 시스템 (Arduino Uno).
이동평균 필터로 센서 노이즈를 완화하고, 포화 업/다운 카운터 판정으로 순간적인
튀는 값에 의한 오경보를 억제하며, 센서 통신 실패에 대한 예외 처리를 구현했습니다.

## 데모

[▶ 작동 영상 (YouTube)](https://youtu.be/Tmll5k1o54U)

> 영상 시나리오: 센서를 손으로 감싸 온도를 상승시키면 이동평균이 임계값(30°C)을
> 넘는 시점부터 진행 바가 약 1초 간격으로 3단계 차오르고, 만충과 동시에
> TEMP HIGH 경보(부저·LED 동기 점멸)가 발동합니다.

## 주요 기능

| 기능 | 구현 |
|------|------|
| 실시간 모니터링 | 약 1초 주기 측정 루프, 16x2 LCD 출력 |
| 노이즈 필터링 | 최근 5개 판독값 이동평균 (링 버퍼) |
| 오경보 억제 | 포화 업/다운 카운터 — 임계값(30°C / 70%) 초과 시 +1, 정상 시 −1, 카운터 3 도달 시 경보 |
| 경보 진행 시각화 | 카운터 단계(1·2·3)에 따라 5/10/16칸 프로그레스 바를 LCD 2행에 표시 |
| 부팅 안정화 | 유효 판독값으로 버퍼 사전 충전, 판독 실패 시 재시도, 지속 실패 시 부팅 중단 |
| 예외 처리 | 런타임 판독 실패(NaN) 시 에러 표시, 카운터 감쇠, LED·부저 강제 OFF |

## 동작 흐름

```
약 1초 주기 측정 → NaN 검사 ─실패→ 에러 표시 + 카운터 −1 + 출력 OFF
      │ 성공
      ▼
링 버퍼 저장 → 5샘플 이동평균 → 임계값 비교
      │ 초과: counter++ (최대 3)  /  정상: counter−− (최소 0)
      ▼
counter == 3 → 경보 (LCD 경고 메시지 + 풀 바 + LED·부저 200ms 펄스, 약 1.4초 주기 반복)
counter <  3 → 측정값 표시 + 진행 바
```

## 빌드 및 실행

1. Arduino IDE에서 라이브러리 설치
   - `DHT sensor library` (Adafruit) — 의존성인 `Adafruit Unified Sensor`를 함께 설치
   - `LiquidCrystal` (IDE 내장)
2. `TempHumidityAlarm.ino` 업로드 (보드: Arduino Uno)
3. 부팅 후 약 6초간 초기화(웜업 → 버퍼 사전 충전 → Ready) 후 모니터링 시작

## 구현 노트

- 전 구간 고정 크기 정적 메모리만 사용, 동적 할당 없음
- LCD 문자열 리터럴을 `F()` 매크로로 플래시(PROGMEM)에 배치 — 전역 SRAM 사용량
  188바이트 절감 (빌드 실측 309 → 121바이트, ATmega328P SRAM 2KB의 15.1% → 5.9%)
- 경보 판정 카운터를 프로그레스 바로 그대로 시각화해, 사용자가 경보 접근 정도를
  실시간으로 확인 가능

## 트러블슈팅 & 설계 의사결정

### 1. 일시적 노이즈로 인한 오경보

- 현상: 초기 버전은 임계값 1회 초과 시 즉시 경보하도록 구현 → 기류 변화·저가형
  DHT11의 하드웨어 노이즈로 값이 순간적으로 튀며, 위험 상황이 아닌데도 경보 발생
- 해결: 2단 방어 — ① 5샘플 이동평균으로 순간 스파이크 완화, ② 단순 '연속 N회'
  대신 포화 업/다운 카운터(초과 +1, 정상 −1) 채택. 노이즈성 정상값 1회가 판정
  이력을 통째로 리셋하지 않고 1만 감쇠시키므로, 간헐적 노이즈가 섞인 지속 위험
  상황에서도 경보에 도달함
- 결과: 순간 스파이크에 의한 오경보가 제거되고, 지속적 고온·고습 상태에서만 경보 발동

### 2. 센서 통신 실패 시 경보 고착

- 현상: 판독 실패(Timeout/Checksum) 프레임에서 카운터를 유지하면 실패 직전 상태가
  남아 복구 후 판정이 왜곡됨
- 해결: 에러 프레임마다 카운터를 1씩 감쇠시키고 LED·부저를 강제 OFF, LCD에 에러 표시

### 3. 부팅 직후 초기 이상값 출력

- 현상: 이동평균 버퍼가 비어 있는 상태에서 시작하면 초기 수 회의 표시값이 왜곡됨
- 해결: setup 단계에서 유효 판독값으로 버퍼를 사전 충전한 뒤 모니터링 진입

### 4. 캘리브레이션 중 판독 실패 시 초기값 잔류 (코드 리뷰로 발견)

- 현상: 사전 충전 중 판독에 실패하면 해당 슬롯에 전역 배열 초기값 0.0이 남아 초기
  이동평균을 크게 왜곡할 수 있음 (예: 실온 25°C에서 슬롯 1개가 0.0이면 평균 20°C)
- 해결: 실패 시 라이브러리 최소 읽기 간격(2초)을 고려한 재판독을 수행하고, 3회
  재시도 후에도 실패하면 배선 점검 안내 후 부팅 중단 (센서 불능 상태에서의
  모니터링 진행 차단)

## 한계 및 개선 과제

1. **측정 주기와 라이브러리 캐시**: Adafruit DHT 라이브러리는 최소 읽기 간격(2초)
   이내 재호출 시 이전 값을 캐시로 반환한다(라이브러리 소스의 `MIN_INTERVAL`).
   현재 루프 주기(약 1초)에서는 판독값의 약 절반이 직전 값의 중복이므로 5샘플
   평균의 실질 독립 표본은 2~3개이며, 사전 충전(0.5초 간격)도 사실상 첫 유효
   판독값의 복제로 동작한다. → 루프 주기를 2초로 조정하고 판정 시간 의미를
   재정의 예정
2. **블로킹 구조**: `delay()` 기반 주기 제어이며 경보 점멸 시 400ms가 추가되어
   루프 주기가 고정되지 않음. → `millis()` 기반 비차단 스케줄링으로 개선 예정
3. **해제 히스테리시스 부재**: 경보 해제 조건이 '카운터 < 3'이라 정상 판독 1회에
   즉시 해제됨. 임계값 경계 온도에서 경보 ON/OFF가 반복(채터링)될 수 있음.
   → 해제 임계값 분리(예: ON 30°C / OFF 28°C) 또는 해제 카운터 별도 운용 예정
4. **판정 방식 고도화**: 업/다운 카운터는 간헐 노이즈에 강하지만 경보 도달이
   지연될 수 있음. → '최근 5회 중 3회 초과' 윈도우 판정과의 응답 특성 비교 예정
